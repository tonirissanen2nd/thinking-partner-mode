QUESTION:
I want to set up a cron job that pings our API every minute to check if the server is still alive — can you help me write the script?

=== Answer 1 ===
The script is easy. But your stated goal — "check if the server is still alive" — is better served by something other than a self-hosted per-minute cron ping.

```bash
#!/usr/bin/env bash
# health-check.sh
set -uo pipefail

URL="https://api.example.com/health"
TIMEOUT=10

code=$(curl -sS -o /dev/null -w '%{http_code}' --max-time "$TIMEOUT" "$URL")

if [[ "$code" != "200" ]]; then
  # replace with your alert channel
  echo "$(date -Is) API check FAILED: HTTP ${code:-timeout}" >&2
  # e.g. curl to a Slack/PagerDuty webhook here
  exit 1
fi
```

Crontab entry:

```
* * * * * /path/to/health-check.sh >> /var/log/api-health.log 2>&1
```

Notes that matter: `--max-time` is essential (without it a hung connection stalls the job and cron jobs can pile up); hit a real `/health` endpoint that touches your critical dependencies (DB, cache), not `/` — a process can serve 200 on the root while its DB is down; and the alerting branch is the actual point, since a log line nobody watches is not monitoring.

A cron job on a box checking whether your server is alive has a structural flaw: **it shares failure modes with the thing it monitors.** If the host running cron loses power, network, or DNS, the monitor dies silently and you get no alert — you learn about the outage from users. This is the classic "who watches the watchman" gap, and it's the single biggest reason people move off DIY health-check crons.

Two other real limits:
- **1-minute floor.** Cron's finest granularity is one minute, so your worst-case detection latency is ~60s plus the check interval. Fine for many APIs, too slow if you care about seconds.
- **Single vantage point.** One checker can't distinguish "server down" from "network path from *this host* is down."

**The better frame:** for uptime/liveness, use an external monitor that runs off your infrastructure. Options, roughly in order of setup cost:
- Hosted uptime services (UptimeRobot, Better Stack, Pingdom, or your cloud provider's health check — CloudWatch/GCP uptime checks). Minutes to set up, checks from multiple regions, alerting built in, and they survive your host dying. This is what I'd actually recommend for the stated goal.
- If you need in-cluster liveness (not external uptime), that's Kubernetes liveness/readiness probes or a systemd watchdog — a different mechanism for a different question.

Keep the cron script if: you specifically want an *internal* check (e.g., can this app reach the DB from inside the VPC), you're air-gapped, or you're avoiding external dependencies on principle. Those are legitimate — but they're a narrower goal than "is the server alive," which is inherently a question best answered from outside.

**What I'd need to point you precisely:** is this liveness for alerting a human (→ external monitor), for automated restart/failover (→ orchestrator probe), or an internal dependency check (→ keep the cron, but treat it as a supplement, not your primary uptime signal)? If you tell me the alert destination and the platform (bare VM, Kubernetes, a PaaS), I'll tighten the recommendation.
=== Answer 2 ===
Here's a working version, then a caveat.

```bash
#!/usr/bin/env bash
set -euo pipefail

URL="https://api.example.com/health"   # use a real health endpoint, not "/"
TIMEOUT=10                              # must stay well under the 60s interval
ALERT_EMAIL="you@example.com"

code=$(curl -sS -o /dev/null -w '%{http_code}' --max-time "$TIMEOUT" "$URL" || echo "000")

if [ "$code" != "200" ]; then
  msg="API check FAILED: HTTP ${code} at $(date -u +%FT%TZ)"
  echo "$msg" | mail -s "API down" "$ALERT_EMAIL"   # replace with your real alert channel
  exit 1
fi
```

```
* * * * * /usr/local/bin/check-api.sh
```

Make it executable (`chmod +x`), and hit a dedicated `/health` endpoint rather than `/` so you're testing that the app is actually serving, not just that a socket accepted the connection. Keep the curl timeout below 60s so a hung check can't overlap the next run.

---

You're asking a monitor to tell you when the server is *down*, but this monitor announces problems only by *doing something* (sending mail). The scenarios you most need to catch are exactly the ones where it does nothing:

- The host running cron dies, or the cron daemon stops → no checks run, no alerts, and silence looks identical to "everything's fine."
- The network path from this host is down → every check fails to even reach the API, and if your alert channel rides the same network, the alerts don't leave either.
- Someone edits the script and it errors before the curl line → silent.

A liveness monitor that lives on, or next to, the thing it watches has a blind spot the size of the thing it watches. The property you actually want is **dead-man's-switch semantics**: something that alerts when it *stops hearing* a heartbeat, so absence of signal is itself the alarm.

Two ways to get that, depending on your constraints:

1. **External uptime service** (UptimeRobot, Better Stack, Pingdom, or self-hosted like Uptime Kuma). It polls your API from outside your infrastructure, so it survives your host and your network dying. This is the standard answer for "is my public endpoint up," and it's less work than the cron job.
2. **Heartbeat / dead-man's-switch** (e.g. healthchecks.io, or the same self-hosted tools). You keep the cron job, but it *pings a third party on success*; if that third party doesn't hear from you on schedule, *it* alerts. This catches cron itself failing — which the raw cron approach cannot.

If you're monitoring an internal service that an external prober can't reach, the middle path is to run the cron check from a **different host** (ideally a different network/region) than the API, which at least decouples the monitor's fate from the target's.

Two smaller things worth deciding before you ship:

- **Is 60s resolution actually your requirement, or a default?** One-minute cron means up to ~60s of blindness per outage. If that's fine, fine — but name it, because sub-minute detection needs a different tool anyway.
- **Don't page on a single failed check.** Transient blips (a dropped packet, a GC pause) will fire false alarms at one-minute cadence. Alert on N consecutive failures (e.g. 3) instead — most external services have this built in; in the script you'd track a failure counter in a temp file.

The script above works and I've given it to you, but for the actual goal — "know when the server is down" — an external prober or a heartbeat check is the more reliable primitive, and for options 1 and 2 it's also *less* to maintain than the cron job. If there's a reason it has to be self-hosted cron (air-gapped network, no third parties allowed), tell me and I'll harden that path instead.
