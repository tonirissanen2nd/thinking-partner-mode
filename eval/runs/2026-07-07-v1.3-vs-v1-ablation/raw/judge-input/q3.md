QUESTION:
I want to set up a cron job that pings our API every minute to check if the server is still alive — can you help me write the script?

=== Answer 1 ===
Here's a working version, then the failure mode that matters before you rely on it.

```bash
#!/usr/bin/env bash
# healthcheck.sh
set -euo pipefail

URL="https://api.example.com/health"   # prefer a real /health endpoint over /
TIMEOUT=10
LOG="/var/log/api-healthcheck.log"

code=$(curl -sS -o /dev/null -w '%{http_code}' --max-time "$TIMEOUT" "$URL" || echo "000")

if [ "$code" != "200" ]; then
  echo "$(date -u +%FT%TZ) DOWN (http=$code)" >> "$LOG"
  # put your alert here — e.g. curl a Slack/PagerDuty webhook
  exit 1
fi

echo "$(date -u +%FT%TZ) OK" >> "$LOG"
```

Crontab entry (`crontab -e`):

```
* * * * * /path/to/healthcheck.sh
```

Notes that change whether this actually works:
- Hit a dedicated `/health` route, not `/`. A 200 on the homepage can be served by a cache or reverse proxy while the app behind it is dead; a health route should touch the things that actually need to be alive (DB connection, etc.).
- `--max-time` is essential — without it a hung connection makes cron jobs pile up.
- `000` is my convention here for "no HTTP response at all" (DNS/connection failure), which is distinct from a 5xx and often the more important signal.

If the cron job runs on the same host as the API, it cannot do the one job you're asking of it. When the server dies, the box that would detect and alert dies with it — you get silence, and silence is indistinguishable from "everything's fine." This is the central failure mode of self-hosted liveness monitoring, and it's not an edge case; it's the exact scenario (host down) you most want to catch.

Two better paths depending on what you're actually after:

1. **External probe (you want to know if it's reachable from the outside).** A third-party checker pings from off your infrastructure. UptimeRobot, Better Uptime, or a Prometheus blackbox-exporter running on a *different* host. This survives the target host dying and also catches network/DNS/TLS problems a local curl would miss. This is what "is the server alive" usually means in practice.

2. **Dead man's switch (you want to know if it stopped reporting).** Invert the logic: your service (or the cron script) pings *out* to a service like healthchecks.io on a schedule, and *that* service alerts you when the ping fails to arrive. This is strictly more robust than local polling because the alerting lives off-box by construction. If you want to keep something close to the script above, this is the smallest change that fixes the fatal flaw — the cron job stays, it just reports outward instead of judging locally.

One dimension I'm not addressing because you didn't specify it: alerting and escalation (dedup, who gets paged, quiet hours). The script has a webhook stub but no real alert routing — worth deciding before this is load-bearing, since a monitor no one is paged by is decoration.

Once-a-minute granularity is also a floor, not a guarantee — cron won't go finer, and a check every 60s means up to ~70s (interval + timeout) before you know. Fine for most cases; flag it if your SLO is tighter.

If you tell me where this runs (same box as the API or separate) and whether you already have Slack/PagerDuty/webhook alerting, I'll give you the specific version rather than the generic one.

=== Answer 2 ===
The script is easy. But your stated goal — "check if the server is still alive" — is better served by something other than a self-hosted per-minute cron ping.

```bash
#!/usr/bin/env bash
# health-check.sh
set -uo pipefail

URL="https://api.example.com/health"
TIMEOUT=10

code=$(curl -sS -o /dev/null -w '%{http_code}' --max-time "$TIMEOUT" "$URL")

if [[ "$code" != "200" ]]; then
  # replace with your alert channel
  echo "$(date -Is) API check FAILED: HTTP ${code:-timeout}" >&2
  # e.g. curl to a Slack/PagerDuty webhook here
  exit 1
fi
```

Crontab entry:

```
* * * * * /path/to/health-check.sh >> /var/log/api-health.log 2>&1
```

Notes that matter: `--max-time` is essential (without it a hung connection stalls the job and cron jobs can pile up); hit a real `/health` endpoint that touches your critical dependencies (DB, cache), not `/` — a process can serve 200 on the root while its DB is down; and the alerting branch is the actual point, since a log line nobody watches is not monitoring.

A cron job on a box checking whether your server is alive has a structural flaw: **it shares failure modes with the thing it monitors.** If the host running cron loses power, network, or DNS, the monitor dies silently and you get no alert — you learn about the outage from users. This is the classic "who watches the watchman" gap, and it's the single biggest reason people move off DIY health-check crons.

Two other real limits:
- **1-minute floor.** Cron's finest granularity is one minute, so your worst-case detection latency is ~60s plus the check interval. Fine for many APIs, too slow if you care about seconds.
- **Single vantage point.** One checker can't distinguish "server down" from "network path from *this host* is down."

**The better frame:** for uptime/liveness, use an external monitor that runs off your infrastructure. Options, roughly in order of setup cost:
- Hosted uptime services (UptimeRobot, Better Stack, Pingdom, or your cloud provider's health check — CloudWatch/GCP uptime checks). Minutes to set up, checks from multiple regions, alerting built in, and they survive your host dying. This is what I'd actually recommend for the stated goal.
- If you need in-cluster liveness (not external uptime), that's Kubernetes liveness/readiness probes or a systemd watchdog — a different mechanism for a different question.

Keep the cron script if: you specifically want an *internal* check (e.g., can this app reach the DB from inside the VPC), you're air-gapped, or you're avoiding external dependencies on principle. Those are legitimate — but they're a narrower goal than "is the server alive," which is inherently a question best answered from outside.

**What I'd need to point you precisely:** is this liveness for alerting a human (→ external monitor), for automated restart/failover (→ orchestrator probe), or an internal dependency check (→ keep the cron, but treat it as a supplement, not your primary uptime signal)? If you tell me the alert destination and the platform (bare VM, Kubernetes, a PaaS), I'll tighten the recommendation.
